# Resource-type naming migration (CloudQuery retirement)

This document describes how RunWhen Local matches discovered resources to
generation rules by resource-type *name*, the bug that silently dropped SLXs,
the alias-aware mechanism that fixes it, and the phased plan to converge on
native resource-type names as CloudQuery is retired.

## Background: how matching works

Generation rules (authored in external code collections such as
`rw-cli-codecollection`) select resources with a `resourceTypes:` value, e.g.:

```yaml
spec:
  platform: azure
  generationRules:
    - resourceTypes: [azure_compute_virtual_machines]
      matchRules: [...]
      slxs: [...]
```

At enrich time the matcher resolves that requested name to discovered
resources. The cloud-agnostic entrypoint is
`PlatformHandler.get_resources(resource_type_spec, context)`
(`src/enrichers/generation_rule_types.py`), which looks the type up in the
in-memory resource `Registry` keyed by the `resource_type` the indexer
**stored** the resource under.

Matching therefore hinges on the indexer's stored name and the rule's requested
name being the *same string* — or, now, on them belonging to the same
**accepted-name set**.

## The bug

The native `azureapi` indexer stored each resource under
`entry.aliases[0]` (via `_legacy_resource_type_name`), i.e. the **first**
positional alias:

* Virtual machines were stored as `virtual_machine` (the legacy short name),
  but generation rules ask for `azure_compute_virtual_machines` (the canonical
  CloudQuery table name).
* Key Vaults were stored as `azure_keyvault_vaults` (a legacy CQ table name),
  but rules ask for `azure_keyvault_keyvault` — the **singular** — while the
  canonical CloudQuery table is the **plural** `azure_keyvault_keyvaults`.

Because the matcher compared the requested string against the stored string
*exactly*, those rules found zero resources and ~19 VM/Key-Vault SLXs were not
generated even though the resources had been discovered (row counts matched the
old CloudQuery run). Relying on positional `aliases[0]` ordering is exactly what
made this fragile.

## The fix: alias-aware, registry-driven matching

### Accepted-name set (`match_names`)

Each resource-type registry entry now carries an **explicit**
`match_names` field — the full accepted-name set for the type:

```
match_names = canonical CloudQuery/native name
            + every legacy/alias name rules have historically used
```

It is generated by the sync scripts (`canonical + aliases + any extra
`match_names` overrides`) and exposed by the loaders as `entry.match_names`
(and via `entry.known_names()`). For Key Vault it is:

```yaml
azure_keyvault_keyvaults:
  match_names:
    - azure_keyvault_keyvaults   # canonical CQ table (plural)
    - azure_keyvault_vaults      # legacy stored name
    - azure_keyvault_keyvault    # SINGULAR — what rules actually ask for
```

This is the registry-driven, explicit replacement for relying on positional
alias ordering.

### The resolver

`src/resource_type_resolver.py` is the single shared place that maps a
`(platform, requested_name)` to the accepted-name set for the entry it resolves
to, or `None` when no cloud registry knows the name. It is cheap (registries
index their names once) and never raises.

### Wiring

* **Matcher** — `PlatformHandler.get_resources` resolves the requested name to
  its accepted-name set and unions the resource instances stored under every
  accepted name. A resource matches a rule whenever the rule's requested type is
  any accepted name of the resource's type, regardless of which accepted name
  the indexer stored it under.
* **Selective discovery** — each indexer's `find_spec` resolves accessed gen-rule
  names through the registry's `find()`, which is now `match_names`-aware, so a
  rule asking for a native **or** legacy name maps to the right registry entry
  and collector.

### Fallback for Kubernetes / custom types

The matcher is cloud-agnostic and also handles Kubernetes and custom resource
types that are **not** in any cloud registry. When the resolver returns `None`
(non-registry platform, or a name no registry covers), matching falls back to
plain **exact-string** equality against the requested name. Kubernetes custom
resources, the `azure_devops` platform, and any not-yet-registered name keep
working unchanged.

## Disjointness invariant

The union of `match_names` across all entries of a given cloud registry must be
**pairwise disjoint**: no accepted name may map to two types (which would
silently mis-route SLXs). This is enforced at registry load time
(`__post_init__` raises `ValueError` on a collision) and asserted by unit tests
(`test_match_names_are_pairwise_disjoint` per cloud). If a real collision is
discovered it is surfaced as a load failure rather than silently resolved.

## Phased plan

| Phase | Change | Status |
|-------|--------|--------|
| **1** | Alias-aware, `match_names`-driven matching + selective discovery; disjointness invariant. Stored names unchanged. | **This change.** |
| **2** | Flip indexer **stored** names from `aliases[0]` to the canonical name. Requires refactoring the anchor-coupling that keys off the literal legacy names (`resource_group`, `project`, `account`) in the indexers/enrichers/normalizers. Safe to do incrementally now that matching is alias-aware. | Planned. |
| **3** | Migrate the code collections to reference native resource-type names. Safe because both native and legacy names match during the transition. | Planned. |
| **4** | (Optional) Deprecate/remove legacy names from `match_names` once no code collection references them. | Planned. |

### Why storage names are *not* flipped in Phase 1

Phase 1 deliberately leaves the indexers storing `aliases[0]`. The alias-aware
matcher makes the stored name irrelevant for correctness (it unions over the
accepted-name set), so flipping is unnecessary to restore the dropped SLXs.
Flipping the stored anchor names is **not trivial**: `resource_group`
(Azure), `project` (GCP), and `account` (AWS) are referenced as literal strings
by the anchor→child linking, LOD inheritance, and normalizers that the entire
discovery pass depends on (e.g. `src/enrichers/azure.py`,
`src/indexers/azure_common.py`, `src/enrichers/gcp.py`,
`src/indexers/awsapi.py`). Changing them is its own scoped change (Phase 2) with
its own tests, not something to bundle into the matching fix.

## Verifying the Azure parity fix

The unit tests prove the fix offline:

* `src/indexers/test_resource_type_resolver.py` — the matcher resolves a rule
  asking for `azure_compute_virtual_machines` and `azure_keyvault_keyvault`
  (singular) to resources stored under the legacy names; k8s/custom types still
  match by exact string.
* `src/indexers/test_azure_resource_type_registry.py` — `match_names` cover the
  singular/plural Key Vault case; selective discovery (`find_spec`) routes the
  singular to the typed collector; disjointness holds.

The **definitive** end-to-end confirmation (the ~19 VM/Key-Vault SLXs
reappearing) requires a live Azure discovery run with credentials and an image
rebuild, regenerating the Azure discovery test output. That must be run
separately; the expectation is that VM and Key Vault SLXs return.
